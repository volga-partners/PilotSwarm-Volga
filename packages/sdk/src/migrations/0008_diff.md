# Diff for CMS migration 0008

Migration file: `cms-migrations.ts` — `migration_0008_session_owner_users`

## Table Changes

### `users` — new

Lazily records authenticated principals when they first create an owned
session.

| Column | Type | Meaning |
|---|---|---|
| `user_id` | BIGSERIAL | Internal stable catalog key |
| `provider` | TEXT | Auth provider ID, for example `entra` |
| `subject` | TEXT | Stable provider subject claim |
| `email` | TEXT | First observed email claim |
| `display_name` | TEXT | First observed display name claim |
| `created_at` | TIMESTAMPTZ | First seen time |
| `updated_at` | TIMESTAMPTZ | Insert time; intentionally not refreshed in phase 1 |

`UNIQUE(provider, subject)` preserves one user row per authenticated
principal.

### `session_owners` — new

One-owner-per-session link table.

| Column | Type | Meaning |
|---|---|---|
| `session_id` | TEXT | Session primary key; no row means system or unowned |
| `user_id` | BIGINT | Linked `users.user_id` |
| `assigned_at` | TIMESTAMPTZ | First assignment time |

`session_id` is the primary key so the first owner assignment wins.

## New Indexes

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_<schema>_users_provider_subject
    ON <schema>.users(provider, subject);

CREATE INDEX IF NOT EXISTS idx_<schema>_session_owners_user
    ON <schema>.session_owners(user_id);
```

The unique user index powers lazy first-seen registration. The owner
index supports future owner-scoped catalog queries without changing the
data model.

## New Stored Procedures

### `cms_register_user(provider, subject, email, display_name)` — new

Registers a user if the `(provider, subject)` pair has not been seen
before, then returns `user_id`.

Uses `ON CONFLICT DO NOTHING` rather than `DO UPDATE`: phase 1 is
first-seen-write-wins and does not refresh profile fields.

### `cms_set_session_owner(session_id, provider, subject, email, display_name)` — new

Registers the user and links the session to that user unless:

- the session is missing
- the session is soft-deleted
- the session is a system session
- the session already has an owner row

The session link also uses `ON CONFLICT DO NOTHING` so the first owner
assignment wins.

### `cms_inherit_session_owner(session_id, parent_session_id)` — new

Copies the parent's owner link onto a non-system child session if the
child does not already have one. This keeps orchestration-spawned child
sessions associated with the top-level authenticated user without
copying owner columns onto `sessions`.

## Function Changes

### `cms_list_sessions()` — return shape changed

PostgreSQL cannot `CREATE OR REPLACE FUNCTION` when the return row shape
changes, so this migration explicitly drops and recreates the function.

```diff
+ owner_provider     TEXT
+ owner_subject      TEXT
+ owner_email        TEXT
+ owner_display_name TEXT
```

The function still reads non-deleted sessions ordered by
`updated_at DESC`, but now left-joins `session_owners` and `users`.

### `cms_get_session(session_id)` — return shape changed

Also drop-and-recreate for the same return-shape reason. It returns the
same owner columns as `cms_list_sessions()`.

## Direct Table Access Guidance

Application code should continue to access CMS through stored
procedures only. The new tables are intentionally implementation
details behind `cms_set_session_owner`, `cms_inherit_session_owner`,
`cms_list_sessions`, and `cms_get_session`.
